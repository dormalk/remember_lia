## Deferred from: code review of 1-4-display-image-slider-with-auto-advance-manual-navigation-and-pause-control (2026-06-09)

- **No WAI-ARIA carousel role / keyboard arrow-key navigation** [src/components/sections/SliderSection.tsx] — the spec mandates keyboard-operable `<button>` elements (implemented) but does not require the full WAI-ARIA carousel pattern (`role="region"`, `aria-roledescription`, arrow-key `onKeyDown`). Revisit as part of the Story 1.9 accessibility polish pass.
- **No `aria-live` region for auto-advance slide changes** [src/components/sections/SliderSection.tsx] — screen-reader users receive no announcement when the slider auto-advances to a new image. Out of story scope; Story 1.9 a11y pass.
- **Auto-advance timer restarts on every tick** [src/components/sections/SliderSection.tsx:118-133] — including `index` in the auto-advance `useEffect`'s dependency array (the mechanism that gives AC2's "reset after manual navigation" its elegant zero-extra-state implementation) has the side effect that the `setInterval` is torn down and recreated on every 5-second tick even during normal automatic playback. No user-visible impact; functionally correct; noted as a micro-optimization opportunity (a `useRef`-based "last manual interaction" approach would avoid the per-tick churn).
- **No visual feedback during in-progress swipe gesture** [src/components/sections/SliderSection.tsx:169-185] — `onTouchStart`/`onTouchEnd` only capture start and end coordinates; no `onTouchMove` tracks in-progress position, so users get no transitional visual cue. Story 1.9 UX polish.
- **Diagonal swipe may trigger unintended horizontal navigation** [src/components/sections/SliderSection.tsx:175-185] — only the X-delta is measured against the 40px threshold; a gesture with large horizontal drift but dominant vertical intent could trigger a slide change. `touch-pan-y` on the container mitigates this at the browser scroll-capture level but not at the JS deltaX threshold level. Full Y-delta discrimination is out of story scope.
- **`aria-pressed` semantic nuance** [src/components/sections/SliderSection.tsx:193-196] — `aria-pressed={playback === "paused"}` means the button is in "pressed" state when paused. Combined with the label changing to "resume," a screen reader announces "Resume button, pressed" — technically correct (the pause feature is engaged) but requiring a non-obvious mental model. The implementation satisfies the spec's explicit requirement for `aria-pressed` + accessible label; Story 1.9 a11y review should evaluate whether a simpler pattern (e.g., single persistent label + state described separately) would read more naturally.
- **No `onError` broken-image fallback** [src/components/sections/SliderSection.tsx:161-165] — a populated `imageUrl` that 404s or fails to load shows the browser's native broken-image icon. Same class as the LogoSection 404 deferral already in this file — closing this gap requires an `onError` state swap to `<EmptyState>`, which should be applied uniformly to all six section components (already a Client Component here, unlike LogoSection, so it's actually simpler here than there). Cross-cutting architectural decision; scope it as a shared concern across sections.
- ~~**`imageUrl: ""` (schema default) reaches `next/image src`**~~ — **RESOLVED in Story 2.4**: `SliderEditor.handleSave()` filters out any entry with empty `imageUrl` (`resolvedSlider.filter((img) => img.imageUrl)`) before calling `saveContent`, so `content.json.slider` never contains an empty-`imageUrl` entry.
